    Reference — what these terms mean
    SPF — Sender Policy Framework
    A DNS record that lists which mail servers are authorised to send on behalf of a domain. The receiving server checks the sending IP against this list.
    pass
    Sending IP is authorised by the domain's SPF record.
    fail
    Sending IP is NOT authorised — the domain explicitly disowns it.
    softfail
    Not authorised, but the domain hasn't requested hard rejection (~all).
    neutral
    Domain publishes no opinion on this IP (?all).
    none
    No SPF record found for the domain.
    temperror
    Temporary DNS failure — result is inconclusive.
    permerror
    Permanent error, e.g. malformed SPF record.
    DKIM — DomainKeys Identified Mail
    A cryptographic signature added by the sending server. The receiving server retrieves the public key from DNS and verifies it. A valid signature confirms the signed headers were not altered in transit.
    pass
    Signature is valid; message was not tampered with for the signed headers.
    fail
    Signature invalid — message may have been modified, or key has been rotated.
    none
    No DKIM signature was present.
    d=
    The signing domain. Should align with the From domain for DMARC to pass.
    s=
    Selector — identifies which DNS key record to look up.
    DMARC — Domain-based Message Authentication, Reporting and Conformance
    Builds on SPF and DKIM. Requires at least one to pass AND the result to "align" with the visible From domain. The domain owner publishes a policy (p=) specifying what receivers should do on failure.
    pass
    At least one aligned mechanism (SPF or DKIM) passed.
    fail
    Neither SPF nor DKIM aligned with the From domain.
    p=none
    Monitor only — no action taken on failure.
    p=quarantine
    Failing messages should be treated as spam.
    p=reject
    Failing messages should be blocked outright.
    dis=
    Disposition actually applied by the receiving server.
    ARC — Authenticated Received Chain
    Preserves authentication results when a message is forwarded or passes through a mailing list. Each relay adds a numbered ARC set so the final receiver can see what authentication looked like before forwarding.
    i=
    Instance number — position in the chain (1 = first forwarder).
    cv=none
    First ARC hop; no prior chain to validate.
    cv=pass
    All previous ARC entries are intact and valid.
    cv=fail
    The chain has been broken — a previous entry was modified.
    SRS — Sender Rewriting Scheme
    Used by forwarding services to rewrite the envelope Return-Path so that SPF passes at the next hop. Addresses follow the pattern SRS0=hash=tt=original-domain=local@forwarder. Its presence indicates forwarding, not spoofing.
    Return-Path (Envelope-From)
    The address to which bounce and delivery-failure notifications are sent. Set separately from the visible From header. Legitimate mail usually has a Return-Path in the same or a clearly related domain as From. A Return-Path in an unrelated domain can indicate spoofing — SRS encoding is a common legitimate exception.
    Reply-To
    Overrides the address used when a recipient clicks Reply. If Reply-To differs from the From domain, replies go to a different address than the apparent sender. This is a common phishing technique used to redirect responses to an attacker-controlled mailbox.
    client-ip
    The public IP address of the connecting SMTP client as seen by the receiving MTA. Set by the receiving server in Authentication-Results and Received-SPF headers — not by the sender — making it the most reliable source of the connecting server's real IP address.
    compauth — Composite Authentication (Microsoft 365 / Exchange Online)
    A Microsoft-proprietary verdict combining SPF, DKIM, DMARC and additional signals into a single result. Appears in Authentication-Results headers on mail processed by Exchange Online Protection.
    pass
    Sufficient authentication signals passed.
    fail
    Explicit authentication failure.
    softpass
    Partial pass — some signals were positive.
    reason=
    Numeric code providing further detail on the result.
    DARA — Domain Alignment Result Assertion (Google)
    A Google-proprietary field indicating whether the DKIM signing domain aligns with the From domain. Appears in Authentication-Results on mail processed by Gmail.
    pass
    Signing domain aligns with the From domain.
    neutral
    Alignment was not evaluated — common in forwarded mail.
    fail
    Signing domain does not align with From.

    Forensic report

    Ready to paste into a technical report or incident ticket. Edit as needed before use.

    All parsing happens locally in your browser — the pasted or dropped header is never uploaded anywhere. Received-header hostnames and "from" fields are self-reported by each relay and can be forged; treat the earliest hop as a lead to investigate, not final proof. Lookalike-domain checks are heuristic (edit-distance against a short list of commonly spoofed brands) and can miss or over-flag — verify manually. No IP geolocation, reverse-DNS, or WHOIS lookups are performed offline.
