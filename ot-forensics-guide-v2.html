Rule zero — process safety outranks evidence

No collection action on a live controller or safety system without the plant/process engineer present and a documented understanding of what the action does to the running process. A mistimed poll, mode switch, or acquisition can trip or damage physical equipment. Capture network-side and volatile host evidence first; gate all controller interaction behind engineering sign-off.

01

Doctrine & ground rules

The five things different enough from IT DFIR that getting them wrong ruins the investigation — or the plant. Everything downstream in this guide follows from these.

Safety & availability dominate
  • The process must often keep running — you rarely get to pull power or take assets offline on demand.
  • A Safety Instrumented System (SIS) is off-limits during response unless its own vendor/engineer drives.
  • Isolation ≠ shutdown. Logically isolate; do not de-energize production without operations' decision.
Volatility is inverted & weird
  • PLC/RTU logic and live state sit in volatile RAM; a reboot or power event destroys them.
  • HMI alarm/event buffers roll in minutes-to-hours on busy systems.
  • EWS project files are the "source of truth" for intended logic — snapshot them early.
Heterogeneous, legacy, proprietary
  • Windows XP/7/Server 2008 HMIs are common; EDR often absent. Bring old, offline-capable tooling.
  • Controllers speak vendor-proprietary engineering protocols with sparse public docs.
  • No agent coverage below L3 — network capture is often your only L1/L0 visibility.
People & process
  • You need a process engineer to interpret whether a setpoint or logic change is malicious.
  • Vendor support contracts gate controller diagnostics & project files — line them up day one.
  • Chain of custody spans OT + safety + regulatory stakeholders. Document relentlessly.
WatchNever run aggressive IT tooling (Nmap SYN sweeps, credentialed vuln scans, WMI storms) against an OT segment. Fragile stacks on old controllers can crash from a single malformed or merely unexpected packet — this has caused real plant trips. Passive-first, always. See ATT&CK ICS techniques and NIST SP 800-82r3.
02

Terrain: mapping evidence to the Purdue model

Orient every asset by Purdue level (ISA-95). It tells you the OS (if any), the protocols crossing it, the volatility of its evidence, and how dangerous it is to touch. Attacker dwell-time and tooling concentrate at L2–L3 (Windows, reachable); physical impact lands at L0–L1.

The controller (L1) is where physical harm is realized — but by the time you look there, the story of how was written at L2–L3.

Collection priority ladder

Order by volatility × value × safety-cost. The five playbooks that follow map 1:1 onto these rungs. Grab volatile, high-value, low-risk evidence first; defer live-controller interaction to the end, behind sign-off.

Why network is #1It's the only free action — no host disturbed, no process touched — and it keeps recording while you work everything else, including controllers you may never be allowed to interrogate directly.
04

Playbook · Passive network capture

Your first action. Non-intrusive, safe, and the highest-leverage single move in OT IR. The goal: get a hashed, timestamped, receive-only capture running at the boundaries that matter, then let it record while you work the hosts.

P1 · Stand up captureRisk: LowATT&CK ICS · TA0102 / TA0101
  1. Pick capture points before touching anything
    At minimum: the IT/OT DMZ boundary (reconstructs the crossing) and a control-network SPAN/TAP (L2↔L1 engineering traffic). If an OT NSM (Nozomi, Claroty, Dragos) exists, pull its existing historical captures and asset inventory first — it may already hold the pre-incident baseline you need.
  2. Prefer a physical TAP over SPAN where fidelity matters
    SPAN ports silently drop frames under load and can miss exactly the low-volume engineering commands you care about. A passive TAP is receive-only by construction — it cannot inject. If you must use SPAN, note it in the record as a fidelity caveat.
  3. Capture receive-only, to disk, with rotation and hashing
    Never let the capture host transmit onto the OT segment (disable the NIC's TX, or use a data-diode/TAP that enforces it). Rotate files so a crash doesn't lose everything; hash each on close. 
    # receive-only capture, 512MB ring files, hash on rotation
dumpcap -i eth1 -w /eth/otf/dmz.pcapng -b filesize:524288 -b files:200
# or tcpdump equivalent
tcpdump -i eth1 -s0 -w /eth/otf/dmz-%Y%m%d-%H%M%S.pcap -G 300 -C 512 -Z root
# immutable hash + timestamp per file (do this as files close)
sha256sum dmz-*.pcap > dmz.sha256; date -u >> capture.log
  4. Baseline "normal" talker pairs
    Before hunting anomalies, extract the conversation map. In OT this is unusually powerful because normal traffic is highly deterministic — the same HMIs poll the same PLCs at the same intervals. Anything off-pattern jumps out. 
    # conversation map + endpoint list from a capture
tshark -r dmz.pcapng -q -z conv,tcp -z endpoints,ip
# who is speaking OT protocols? (S7=102, modbus=502, enip=44818)
tshark -r ctrl.pcapng -Y "s7comm || modbus || enip || dnp3" -T fields \
  -e ip.src -e ip.dst -e _ws.col.Protocol | sort | uniq -c | sort -rn
  5. Flag the OT-specific red patterns
    Use protocol-aware dissectors. Look for write/operate commands from a host that normally only reads, controller STOP/PROGRAM-mode or download commands outside a maintenance window, forced-I/O, new devices speaking engineering protocols, and malformed/scan-like frames. For volume, run Zeek with ICS parsers (Amazon/spicy Modbus, S7, DNP3, ENIP) to get structured logs instead of packet-by-packet grinding.
ToolsWireshark / tsharktcpdump / dumpcapZeek + ICS parsersArkimeMalcolm (CISA)NetworkMinerNozomi / Claroty / Dragos exportsphysical TAP
Do notDo not "just run Nmap to map the segment." Discovery scans are an active injection onto a fragile network. Derive your asset map from the passive capture instead.
05

Playbook · Volatile state on L2/L3 Windows hosts

HMIs, SCADA servers, and especially the engineering workstation. These run Windows, so standard live-response applies — with the twist that you often can't take them offline, EDR is usually absent, and the OS may be ancient. Prioritize the EWS: it holds the project files and the software that can push logic to L1.

P2 · Live-response an OT Windows hostRisk: MedATT&CK ICS · Execution / Persistence
  1. Decide isolation without shutdown
    With operations, choose logical isolation (port ACL, VLAN move, or unplugging an uplink — never the process-facing link mid-operation) over power-off. Document the decision and who authorized it. A rebooted HMI loses volatile state and may blind operators to the process.
  2. Capture RAM first, to external media
    Memory holds injected code, decrypted payloads, C2 handles, and credentials — and it's the most volatile persistent-value evidence on the box. Write to an external drive, never to the host's own disk. 
    # small, offline-capable acquirers work on old Windows
winpmem_mini.exe E:\otf\HMI01.raw
# Magnet RAM Capture / DumpIt are GUI alternatives for XP/7 era
certutil -hashfile E:\otf\HMI01.raw SHA256 >> E:\otf\hashes.txt
  3. Grab live network + process state
    Before killing connections or the box, snapshot what's talking and running. On old hosts, prefer statically-linked tools you brought, not the host's binaries. 
    # connections, processes, services, scheduled tasks, sessions
netstat -anob     tasklist /v /svc
schtasks /query /fo LIST /v     query user
# or one shot with a known-good toolkit (KAPE / velociraptor / SysinternalsSuite)
  4. Triage-collect the standard Windows artifact set
    Event logs (winevt\Logs\ — 4624/4625 logons, 7045 service installs, RDP/TerminalServices), Prefetch, registry hives + NTUSER.DAT (Run keys, ShimCache/AmCache, USB), scheduled tasks, and RMM footprints (AnyDesk/TeamViewer/ScreenConnect abused into OT). KAPE or Velociraptor pull all of this quickly and consistently. 
    # KAPE targeted triage to external drive
kape.exe --tsource C: --target !SANS_Triage --tdest E:\otf\HMI01 --vhdx HMI01
  5. Note what's OT-abnormal on this host
    Engineering software running under a non-engineer account or off-hours; RMM freshly installed; new local admin on an HMI; logic-project files with modified timestamps outside change control. These become your pivots into the EWS and controller playbooks.
ToolsWinPMEM / Magnet RAM / DumpItKAPEVelociraptorSysinternalsFTK Imager (live)Eric Zimmerman tools
Order within the hostRAM → live net/process state → triage artifacts → (later) full disk image. If forced to choose one thing on a box you can't keep, take RAM.
06

Playbook · OT logs, alarms & engineering audit trails

The OT-specific evidence IT DFIR misses — and much of it rolls fast, so it sits at priority #3 ahead of full disk imaging. This is where you find who pushed logic and when, and whether operators were blinded.

P3 · Export the OT audit trail before it rollsRisk: LowATT&CK ICS · T0804 / T0821 / T0835
  1. HMI/SCADA alarm & event history
    The operator-visible timeline of process anomalies. Export it now — busy systems overwrite in minutes-to-hours. Suppressed or disabled alarms are a headline finding (operators blinded to a manipulated process). Storage is vendor-specific: Wonderware/AVEVA InTouch & WinCC log to SQL or flat files; Ignition to its internal DB; FactoryTalk View to its own event DB. Export via the product's tooling to a static file, and hash it.
  2. Engineering-software logs — the smoking gun
    These record downloads/uploads to controllers: the who/when of logic changes. A download timestamp matching your intrusion window, from an account that shouldn't do engineering, often is the case.
    Vendor stackWhere to look
    Rockwell / FactoryTalkFactoryTalk Diagnostics logs; RSLinx/FactoryTalk Linx comms logs; Studio 5000 activity
    Siemens TIA / Step7TIA Portal logs; the controller's own diagnostic buffer (mode changes, downloads)
    Schneider (EcoStruxure/Unity)Control Expert logs; OFS/comms driver logs
    OPC layerKEPServerEX / Matrikon / OPC UA server logs — connection & write history
  3. Windows event logs on the OT host, for correlation
    Tie the engineering-software events to logon/session events (4624/4625, RDP, RunAs) so you can attribute the logic push to a session, source host, and account. This bridges the OT audit trail to your host timeline.
  4. Historian recent data (hand-off to §08)
    Recent, high-resolution historian data also rolls or downsamples. Kick off the tag export now (detailed in the historian playbook) so the raw values covering the incident window are frozen before the historian aggregates them.
  5. Preserve everything as static, hashed exports
    Alarm DBs and logs are live and mutable. Export to flat files, hash, and record the export method and time in the chain of custody — you're evidencing a query against a running system, so note that explicitly.
Interpretation moveThe three logs that matter most — engineering download log, HMI alarm/event history, historian tag values — get overlaid in §08 to locate both the action and its concealment.
07

Playbook · EWS disk image & project-file analysis

The engineering workstation is the highest-value host in most OT intrusions: it holds the authoritative project files and the software capable of reprogramming controllers. Persistent evidence, so it sits at priority #4 — but the project-file snapshot should happen early (it's your baseline for the controller diff in §09).

P4 · Image the EWS and secure the "source of truth"Risk: MedATT&CK ICS · T0873 / T0839 / T0843
  1. Snapshot the engineering project archive first
    Before the full disk image, copy the project files and any local version-control/backup folders to external media and hash them. These define the intended control logic and are what you'll diff the running controller against. 
    # typical project-file extensions to hunt for
Rockwell:  *.ACD *.L5X *.L5K      Siemens: *.ap1x *.s7p *.zap1x
Schneider: *.stu *.xef *.sta       GE:      *.lgx
HMI:       *.gwbk (Ignition)  InTouch/WinCC/FTView app dirs
  2. Full forensic disk image (dead or live)
    If the box can be taken down safely, dead-image it. If it must stay up (common), take a live image. Hash source and image; verify. 
    # live image to external drive, verified
FTK Imager → Create Disk Image → Physical Drive → E:\otf\EWS01.E01 (verify hashes)
# linux equivalent
ewfacquire -t /mnt/ext/EWS01 -d sha256 /dev/sda
  3. Reconstruct the engineering activity timeline
    From the image: project-file MAC times, ShimCache/AmCache/Prefetch for the engineering software and any attacker tooling, jumplists/LNK for recently opened projects, and the comms-driver/OPC config that maps which controllers this EWS can reach. Cross-reference against the §06 download logs.
  4. Diff project files against known-good
    Compare the on-disk project to the last trusted backup / version-control tag. Look for edits outside change management, added routines/function blocks, altered setpoints or scaling, and modified HMI scripts that misrepresent process state. Vendor engineering software can produce a logic diff; export it as evidence.
  5. Establish the known-good baseline for §09
    The trusted project file becomes the reference for the live-controller upload-and-compare. If no trusted backup exists, that gap is itself a finding — and raises the bar for interpreting the controller state.

08

Playbook · Historian export & process timeline reconstruction

The historian (OSIsoft/AVEVA PI, AVEVA/Wonderware Historian, GE Proficy, Canary) is the time-series record of the physical process and a frequent pivot, since it straddles the IT/OT DMZ. It's both evidence of what the process actually did and a host to forensicate.

P5 · Export tags, then build the fused timelineRisk: LowATT&CK ICS · T0831 / T0832 / T0836
  1. Export the relevant tags across the incident window
    Pull raw (not downsampled) values for the process tags in scope — setpoints, PVs, actuator commands, key interlocks — for the incident window plus a clean baseline period on either side. Export to CSV/flat file and hash. The baseline lets you distinguish "abnormal" from "normal operating envelope."
  2. Forensicate the historian host itself
    It's usually Windows + SQL: treat it as an L2/L3 host (§05) and a database. Check SQL logs, linked servers, and xp_cmdshell abuse; review the historian's own audit log for who queried, modified, exported, or backfilled data — edited history is cover-up evidence.
  3. Map the historian's reach
    Its data-collection interfaces (OPC, protocol connectors) reach toward controllers. That config both explains reachability and is a manipulation surface — a tampered interface can feed operators false values while the plant does something else.
  4. Overlay three timelines to find action + concealment
    This is the analytical payoff of the whole collection effort: 
    # fuse on a common time axis
[A] engineering download log   # when logic changed, by whom
[B] HMI alarm / event history  # what operators were shown
[C] historian tag values       # what the process actually did

# the tell:
A shows a logic download at T0
C shows a process excursion beginning ~T0
B shows NO alarm for that excursion  ← manipulation + concealment
    Where a logic download precedes a process excursion that the alarms didn't surface, you've found both the action and the operator-evasion.
ToolsPI DataLink / SMT exportvendor historian exportSQL log reviewpandas / Excel for overlaytimeline (plaso/log2timeline)
09

Playbook · Controller collection (L1 — GATED)

Specialist, safety-gated, last. Controllers hold their program and live state in volatile memory, speak proprietary protocols, and can trip the process if interrogated wrong. Default posture: observe via the §04 capture; interrogate the controller directly only with engineering + vendor present and sign-off recorded.

P6 · Interrogate a controller (only if authorized)Risk: HIGH — gatedATT&CK ICS · T0821 / T0839 / T0843 / T0800
  1. Get explicit engineering + vendor sign-off, in writing
    Every action below is a controller interaction that can affect the running process. Confirm current mode, criticality, and what each planned action does. If the controller feeds an SIS, the SIS vendor drives. Record who authorized what.
  2. Capture non-invasive state first
    Read what you can without changing controller state: key-switch/mode position (RUN / PROGRAM / REMOTE — a controller in REMOTE/PROGRAM when it should be RUN is both a finding and an exposure), firmware version, and the diagnostic/fault buffer (e.g. Siemens diagnostic buffer, controller fault logs recording mode changes, downloads, STOP/RUN transitions).
  3. Upload the running logic and diff vs. known-good
    Using the vendor engineering software, upload (read) the live program and compare to the trusted project file from §07. The upload is itself a controller interaction — engineering must approve timing. Look for rogue rungs/routines, added function blocks, altered setpoints or scaling, and logic that doesn't match the documented design.
  4. Capture volatile controller state
    Current register/tag values, force tables (forced I/O is a classic manipulation — an actuator driven regardless of logic), and timer/counter state. All lost on power cycle, so capture before any reset is contemplated.
  5. Firmware integrity (vendor-assisted)
    Compare firmware version and, where extractable, hash against the vendor baseline. Malicious/backdoored firmware is rare but high-impact; extraction is deeply vendor-specific and usually needs vendor support. Do not attempt improvised dumping on a live safety-relevant controller.
  6. Only then consider remediation state changes
    If the controller must eventually be reset or replaced, ensure full state is captured first, with sign-off. Sequence the change with operations so the process is in a safe state.
NeverNever power-cycle a controller "to get a clean state," never switch its mode for convenience, and never push it into a diagnostic mode casually. Each can destroy volatile evidence and disrupt the process. When unsure, stay on the network side (§04).

What "bad" looks like — OT-specific indicators

Beyond standard host/network IOCs, these patterns are distinctive to OT intrusions. Map each finding to ATT&CK for ICS for structured reporting.

On hosts (L2/L3)
  • Engineering software launched by, or logic pushed from, a non-engineer account or off-hours.
  • Modified HMI scripts/screens that misrepresent process state to operators.
  • Project files edited outside change-management; timestamps inconsistent with the version-control record.
  • RMM tools (AnyDesk, TeamViewer) installed on OT hosts; new local accounts on HMIs.
  • Disabled/suppressed alarms in HMI config.
On controllers / process (L0/L1)
  • Running logic ≠ known-good project file (rogue rungs, added blocks, altered setpoints).
  • Forced I/O / forced values in the controller's force table.
  • Unexpected RUN/STOP/PROGRAM transitions in the diagnostic buffer.
  • Process values physically implausible, or frozen while the plant actually moved (sensor spoofing).
  • Firmware version/hash mismatch vs. vendor baseline.
On the network
  • New device speaking an engineering protocol; write/operate from a normally read-only host.
  • Control-protocol traffic crossing the IT/OT boundary that policy should block.
  • Beaconing from an OT host to external/DMZ infrastructure.
  • Scanning or malformed frames aimed at controllers.
Known OT malware shape (context)
  • Frameworks that speak ICS protocols natively to issue control commands (Industroyer/CRASHOVERRIDE, TRITON/TRISIS, PIPEDREAM/INCONTROLLER lineage).
  • Tooling targeting the safety system specifically (TRITON hit a Triconex SIS).
  • Wipers timed to a physical event; logic-bomb scheduled tasks on the EWS.
  • Living-off-the-land via legitimate engineering software rather than custom malware.

Worked scenarios — the chain end-to-end

Two condensed walkthroughs showing how the playbooks connect. These are illustrative composites, not any specific real incident.

Scenario A · Unauthorized logic change on a Rockwell PLC

Trigger: Operators report a pump behaving oddly; an EDR alert fired on a jump host in the IT/OT DMZ two days earlier.

  • §04: Stand up a passive TAP at the DMZ and a SPAN on the control VLAN. Existing Dragos sensor yields a 30-day baseline. The capture shows CIP write/forward-open traffic to the PLC from the EWS at 02:14 — outside any maintenance window.
  • §05: Live-response the EWS. RAM capture + KAPE triage. RDP logon (4624, type 10) at 02:07 from the DMZ jump host, under a maintenance account that shouldn't do engineering. Studio 5000 in Prefetch/AmCache with an execution time matching.
  • §06: Export FactoryTalk Diagnostics — a logic download to the PLC at 02:14, tying the session to the controller write seen on the wire.
  • §07: Snapshot the .ACD project and diff vs. last version-control tag: an added rung forcing the pump output regardless of the level interlock.
  • §08: Historian shows the tank level climbing past setpoint from ~02:15; the corresponding high-level alarm was disabled in the HMI — concealment.
  • §09 (gated): With engineering, upload the running logic → matches the tampered project, and the force table shows the pump output forced. Capture state, then remediate with operations.
  • ATT&CK ICS: map to Modify Program, Program Download, Modify Parameter, Alarm Suppression, Loss of Control. Entry → pivot → control action → concealment reconstructed.
Scenario B · Suspected historian tampering feeding false readings

Trigger: Product quality drifted out of spec, but the historian trends look nominal — a mismatch between reality and the record.

  • §04: Passive capture at the DMZ shows an OPC UA session to the historian from an unexpected host in the DMZ, plus periodic beaconing from that host outbound.
  • §08 (host): Forensicate the historian (Windows + SQL). Audit log reveals backfilled/edited historical values over the drift window; SQL logs show writes from the OPC session, not the normal collector.
  • §08 (tags): Export raw tags for the affected loop plus a clean baseline. The raw upstream values differ from the stored/displayed values — the record was overwritten to mask the real process.
  • §06: HMI alarm history for the same window is clean because operators were shown the falsified series — a second concealment layer.
  • §05/§07: Trace the OPC host: RMM tool freshly installed, C2 beacon in RAM, lateral path from the DMZ jump host. Image it; recover tooling.
  • ATT&CK ICS: Spoof Reporting Message, Rootkit/Manipulation of View, Point & Tag Identification, Loss of View for operators. Physical impact explained by the concealed manipulation.
